Repository, commit, push: three different things

People use these words as if they mean the same action. They do not. One is a place that stores history, one saves a change into that history, and one sends saved history to another machine. Follow a single edit through all three.

repoRepository

A place, not an action. The project folder together with its complete history of every commit. The copy on your laptop is the local repository. The copy on GitHub is the remote repository. Same history, two locations.

git init

commitCommit

An action that saves a snapshot of your changes into your local repository. It is a labelled checkpoint in history. Nothing leaves your laptop.

git commit -m "Add login page"

pushPush

An action that sends commits you already made from your local repository up to the remote one on GitHub. This is the step that shares your work with everyone else.

git push

The one-line version. Commit early and often on your own laptop. Then push when you want your teammates to have those commits too. You can make ten commits and push them all at once.

Commands and skills: two ways to save your work

Both let you teach Claude something once and reach for it again. A command is a shortcut you type for a prompt you would otherwise write out every time. A skill is a whole workflow, several steps plus any scripts or templates it needs, bundled in a folder Claude can run for you.

Command

A prompt you type

.claude/commands/format-refs.md

one file
you type /format-refs
"Format the references I paste in ACS style."
one saved instruction

Skill

A workflow Claude runs

.claude/skills/publications-page/

SKILL.md fetch.py template.html exclusions.json
you type /publications-page or just ask, and Claude runs it when it fits
  1. confirm the ORCID is the right person
  2. fetch the papers from OpenAlex
  3. flag any that may not be theirs
  4. build the page from the template
many steps, plus its own files

commandCommand

A saved prompt in .claude/commands/. You type /name and Claude runs that one instruction. Reach for it when you keep asking for the same single thing.

/format-refs

skillSkill

A folder in .claude/skills/ holding a SKILL.md plus any scripts and templates the job needs. It carries a multi-step workflow. Type /name to run it, or just describe the job and Claude reaches for it on its own.

/publications-page

The rule of thumb. If you would type the same prompt again, save it as a command. If it is a workflow with steps, and it needs scripts or templates to carry along, make it a skill. Both are things you can run by name with /, and a skill has one extra trick: Claude can reach for it on its own when it fits. A rule is different again: you never invoke it, it is always-on guidance Claude follows on its own, like the commit rule you wrote earlier.
